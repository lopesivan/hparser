Goals of the hparser Project

NOTE: This hasn't been updated in a while, although it's still technically accurate. I'll be revising this with better composition and detail sometime soon. - Kevin Barry

The hparser project is a set of Linux shared libraries intended to help programs create their own unique file formats and to deal with external file formats. I haven't been able to pin down solid documentation yet; however, I have managed to define several definite goals for the project. This project is still fairly new, but is nearly completely functional, really only lacking documentation and standard implementation procedures.

Long-term goals of the hparser project:

  1. Allow simple implementation of a parser program or library which can parse an arbitrary input type, in an arbitrary format, and provide simple means to assemble the data into a tree.

  2. Provide simple means for a program to arbitrarily modify the structure of a data tree without additional function calls to modify a parallel-operating interface.

  3. Allow a program to create a fully-functional data tree without parsing input data.

  4. Provide simple means for a program to export a tree's data in a format which can reproduce the tree in its exact state using the same parser.

  5. Allow a program to compile parsed information or tree data into an arbitrary representation.

  6. Provide a simple application interface which allows input from sources, output to destinations, and working data of types unknown to the library.

  7. Allow fully-functional programs to be composed strictly of an interface-only program (such as a GUI) linked to a single library which contains all importation, processing, and exportation functionality.

  8. Allow a fully-linked program which contains all necessary logic except for a file format, which can be arbitrarily changed by changing format libraries.

  9. Retain a fully-functional and safe data tree in the event parsing encounters an error, which also allows simple return to the point of error if the program chooses to continue parsing.

  10. Provide safe and complete memory management which prevents memory leaks, bad pointers, and double deletion.

  11. Allow simple extension of the core library and provide a few basic extensions for common uses.

  12. Provide all above listed functionality with the option of centralized, decentralized, or recursive parsing and exportation, or any combination thereof.

Intended and supported implementations of the hparser project:

  1. Create arbitrary file formats for programs which need to store, load, and modify their own files.

  2. Convert between different file types and formats.

  3. Communicate structured information between independent processes.

  4. Create an arbitrary interpreted language.

  5. Graphically create and modify files.

Possible uses of the hparser project which aren't its best uses and aren't specifically supported:

  1. Parse information which will be digested but will never need to be represented as a tree in run-time memory.

  2. Compile source code which is better compiled by something specifically designed for that task.

hparser currently supports all of its main goals in theory, although nearly all of them have been tested and verified. Documentation of how to implement the libraries will follow soon after I decide how best to communicate it. Until then, please take a look at the download page and the example code it contains.
